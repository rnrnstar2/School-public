# 人対応への切り分けを設計する

ECサイト（＝インターネット上のお店）を運営していると、毎日お客様から問い合わせが届きます。「注文を変更したい」「商品が届かない」「返品したい」——内容はさまざまです。

すべての問い合わせを自分ひとりで受け止めていたら、いつまで経っても終わりません。そこで役立つのが、AIツールを使った自動応答です。

でも、**すべてをAIに任せてよいわけではありません。** お客様が怒っている時や、複雑な事情がある時は、人間のスタッフが直接対応したほうがお客様も安心します。

このレッスンでは、「どの問い合わせをAIに任せ、どれを人間が対応するか」の**切り分けルール表**を、AIツールの力を借りながら設計します。完成すると、チームの誰が見ても同じ判断ができる1枚のドキュメントが手に入ります。15分で仕上げましょう。

---

## イメージをつかむ

まずは冷蔵庫の食材整理を想像してみてください。

- **そのまま食べられる**（サラダ、ヨーグルト）→ すぐ出せる＝**AI対応でOK**
- **火を通す必要がある**（生肉、生魚）→ ひと手間かける＝**人間が対応**
- **賞味期限が近い**（今日中に食べないとダメ）→ 最優先で判断＝**緊急は即・人間へ**

問い合わせも同じように仕分けると、どれを急いで対応すべきか一目でわかるようになります。

![問い合わせ切り分けフロー](/lesson-assets/atom.ec-operator.design-escalation-rules/diagram.png)

上の図のように、問い合わせが届いたら「定型？」→「感情的？」→「例外？」の順に判断するだけで、AI対応か人間対応かをすばやく決められます。

---

## 準備する

始める前に、以下を手元に用意してください。

- **AIツール**: ChatGPT、Claude、またはお好みのAIチャットツール（ブラウザで開いておく）
- **過去の問い合わせ**: 直近1週間〜1ヶ月分をざっと見返せる状態にする（メール受信箱やショップ管理画面など）
- **メモ用のテキストエディタ**: Googleドキュメント、メモ帳、Notionなど、表が作れるものなら何でもOK

---

## ステップ1: 問い合わせの傾向を洗い出す

まずは、あなたのショップにどんな問い合わせが来ているかを書き出します。過去の問い合わせを振り返り、大まかなカテゴリを5〜8個に分類しましょう。

AIツールに次のプロンプト（＝AIへの指示文）をコピペして送ります。

```
ECサイトのカスタマーサポートに届くお客様からの問い合わせを、よくあるカテゴリに分類してください。
カテゴリ名と、それぞれに含まれる具体的な問い合わせ例を2〜3個ずつ箇条書きで出力してください。
```

AIの出力結果を見たら、あなたのショップの実情に合わせて調整します。

### 良い例

「AIが出した『ギフト包装』カテゴリはうちにはないから削除して、代わりに『定期購入の変更』を追加しよう」

### 悪い例

「AIが出したものをそのまま全部使おう」→ あなたのショップに来ない問い合わせまで含まれてしまい、使えないルール表になります

---

## ステップ2: 切り分けルールの表を作る

カテゴリが洗い出せたら、次は「AI対応」か「人間対応」かを決めるルール表を作ります。

AIツールに以下のプロンプトを送りましょう。`【 】`の中はあなたのカテゴリに置き換えてください。

```
以下のECサイト問い合わせカテゴリについて、「AIが自動で応答してよい条件」と
「人間のスタッフに引き継ぐべき条件」を表形式で整理してください。

カテゴリ：【配送、返品・交換、商品不具合、支払い、定期購入】

各カテゴリについて、条件はできるだけ客観的な基準（例：〇日以上経過、
「怒」「不満」などの言葉が含まれる）で書いてください。
```

ポイントは「**客観的な基準で書いて**」とAIに指示することです。これにより「状況に応じて」のような曖昧な表現ではなく、誰が見ても同じ判断ができる条件が出てきます。

出力された表をベースに、あなたのショップならではの条件を足したり削ったりして仕上げます。

### 良い例

| カテゴリ | AIで対応する条件 | 人間が対応する条件 |
| --- | --- | --- |
| 配送 | 「配送状況を教えて」などステータス確認のみ | 「届かない」かつ発送から3日以上経過 |
| 返品・交換 | 返品期限・手順の案内（期限内） | 期限切れ、または「怒」「不満」等の感情的な表現あり |
| 商品不具合 | —（すべて人間対応） | 不具合報告はすべて人間が確認 |
| 支払い | 支払い方法・手順の案内 | 二重請求、返金要求 |

### 悪い例

| カテゴリ | AIで対応 | 人間が対応 |
| --- | --- | --- |
| 配送 | たまにAIで | たまに人間で |
| 返品 | 簡単なもの | 難しいもの |

→ 「たまに」「簡単」「難しい」では判断基準になりません。**数字や具体的なキーワード**で条件を書きましょう。

## ステップ3: 迷ったときのデフォルトルールを決める

どんなに丁寧にルールを作っても、どちらとも判断しにくいケースは必ず出てきます。そのための「迷ったらこうする」というデフォルトルール（＝初期設定のルール）を1つ決めておきましょう。

おすすめは、**「迷ったら人間に回す」** です。

AIが間違った対応をしてお客様を怒らせるリスクと、人間が少し余分に対応するコストを比べると、後者のほうがずっと安全です。運用しながら「これはAIでよかったな」と分かったものを徐々にAI対応に移していけばOKです。

---

## ステップ4: 過去の問い合わせでテストする

ルール表ができたら、実際に使えるか検証します。過去の問い合わせを3〜5件ピックアップして、ルール通りに分類できるか試しましょう。

AIツールにこんなプロンプトを送ると、テストを手伝ってもらえます。

```
以下の切り分けルール表を使って、この問い合わせが「AI対応」「人間対応」の
どちらになるか判定してください。判定の理由も教えてください。

【ルール表をここに貼る】

問い合わせ例：
1. 「3日前に注文しましたがまだ届きません」
2. 「届いた靴の左右で色が違います。写真を送ります」
3. 「返品の手順を教えてください」
```

### テスト結果の読み方

- **正しく分類できた** → そのルールはOK ✓
- **AIツールも迷った** → ルールの条件が曖昧なサイン。条件をより具体的に書き直す
- **想定と違う分類になった** → ルール表に漏れているパターンがある。カテゴリまたは条件を追加する

### 良い例

- 「3日前に注文したがまだ届かない」→ 配送カテゴリ → 発送から3日以上 → **人間対応** ✓
- 「届いた靴の左右で色が違う。写真を添付します」→ 商品不具合 → **人間対応** ✓
- 「返品の手順を教えてください」→ 返品・交換 → 手順案内 → **AI対応** ✓

### 悪い例

- 同じ「届かない」という問い合わせなのに、判定する人によってAI対応になったり人間対応になったりする → ルールの条件が曖昧です。「発送から◯日以上」のように数字で線引きしましょう

---

## ステップ5: 成果物をまとめて保存する

テストが終わったら、最終的な切り分けルール表を1枚のドキュメントに整理しましょう。AIツールに次のプロンプトを送ります。

```
以下の切り分けルールを、チームで共有できる見やすいドキュメントに整理してください。

含める内容：
- ルール表（カテゴリ / AI対応条件 / 人間対応条件）
- デフォルトルール（迷った場合の対応方針）
- 最終更新日

【ルール表の内容をここに貼る】
```

出力されたドキュメントを、Googleドキュメントやチームの共有フォルダに保存すれば完成です。

---

## 確認する

以下の3つをチェックして、すべてOKならこのレッスンは完了です。

- [ ] **問い合わせカテゴリが5つ以上**あり、自分のショップの実情に合っている
- [ ] **各カテゴリに「AI対応の条件」と「人間対応の条件」**が客観的な基準で書かれている
- [ ] **デフォルトルール**（迷ったときの対応方針）が明記されている

完成したルール表のスクリーンショットを撮って保存しておきましょう。これがこのレッスンの成果物です。

## つまずきポイント

- **ルールを一度に完璧にしようとしない**: 最初はざっくり5カテゴリで十分です。運用しながら追加していく前提で大丈夫です
- **AIに全部任せようとしない**: クレーム・感情が絡む案件をAIだけで処理すると、かえってお客様を怒らせることがあります。「迷ったら人間へ」を基本にしましょう
- **条件をあいまいにしない**: 「怒っている場合」ではなく「『不満』『怒り』『二度と買わない』等のキーワードが含まれる場合」のように、客観的な判断基準を意識して書きましょう
- **AIの出力をそのまま使わない**: AIが作った表はあくまで「たたき台」です。あなたのショップの事情を知っているのはあなただけ。必ず自分の目で確認・修正してから使いましょう
- **テストを飛ばさない**: 「表ができたから終わり」ではなく、必ず実際の問い合わせで試してください。テストで初めて見つかる抜け漏れが必ずあります
