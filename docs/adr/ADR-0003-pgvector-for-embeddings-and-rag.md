# ADR-0003: 埋め込み検索と RAG に pgvector を採用する

- Status: Accepted
- Date: 2026-04-08

## Context

MVP では、会話ログ、レッスン本文、改善メモなどを対象に埋め込み検索と RAG を行いたい。用途は主に次の 2 つである。

- 学習者との会話履歴やペルソナ情報を踏まえた文脈取得
- レッスン資産の意味検索と近傍参照

一方で、MVP は Supabase 中心で組み、インフラを増やさない方針である。また、検索時には単純なベクトル近傍だけでなく、公開状態、ペルソナ、言語、バージョンなどのリレーショナル条件で絞り込みたい。

## Decision

ベクトルストアは Supabase Postgres の `pgvector` 拡張を採用する。

- 会話ログ埋め込みとレッスン埋め込みは Postgres 内に保持する。
- ベクトル検索は SQL フィルタと組み合わせたハイブリッド検索で行う。
- 埋め込み生成はジョブもしくは Owner ローカルのバッチ処理から投入する。
- 検索対象ごとにメタデータ列を持たせ、`persona_id`、`lesson_version_id`、`language`、`published_at` などで絞り込む。
- RAG 取得は「候補抽出 -> SQL フィルタ -> ランク付け」の単純な流れに留める。

## Consequences

- 埋め込みデータと元データが同じ DB にあるため、一貫性、バックアップ、権限管理が単純になる。
- SQL だけで「類似度 + 公開状態 + ペルソナ制約」を同時に扱える。
- 追加ベンダーが不要なので、MVP のコストと運用が軽い。
- 一方で、専用ベクトル DB と比べると超大規模コーパスや高度な ANN 最適化では不利になる。
- インデックス設定やクエリチューニングは Postgres 側で意識して行う必要がある。
- コーパス規模や検索負荷が将来増えた場合でも、まずは `pgvector` のまま計測し、限界が見えてから切り出せばよい。

## Alternatives Considered

### Pinecone

不採用。専用ベクトル DB として扱いやすいが、MVP で新しい SaaS を増やす必要がある。元データと埋め込みの同期、権限整合、障害時の切り分けが二重化し、現段階ではコストに見合わない。

### Weaviate

不採用。検索機能は豊富だが、データモデルを別系統で持つことになり、Supabase 中心の構成から外れる。GraphQL/専用 API 学習コストも増え、MVP で必要な範囲を超える。

### Qdrant

不採用。性能面では十分だが、自己運用か別マネージド基盤が必要になる。ベクトル検索のためだけに新しい運用面を増やすより、まずは Supabase 拡張で一貫性を優先する。
