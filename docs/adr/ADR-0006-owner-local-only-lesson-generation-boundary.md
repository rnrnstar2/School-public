# ADR-0006: レッスン生成と改善は Owner ローカル実行に限定する

- Status: Accepted
- Date: 2026-04-08

## Context

この MVP の差別化要素は、疎結合な Lesson Atoms、ペルソナ別 Anchor Flows、制約付き Plan Compiler に加えて、レッスン生成と改善の質を Owner が担保する点にある。Owner（daichi）は Claude Code サブスクと MCP を使ったローカル実行環境を持っており、現時点ではこの編集ループをサーバー自動化へ移す必然がない。

同時に、次の懸念がある。

- サーバー側で自動生成や自動改修まで行うと、品質劣化や意図しないカリキュラムドリフトを招きやすい。
- LLM 利用量と外部ツール権限をサーバーに集約すると、コストとセキュリティ境界が膨らむ。
- MVP 段階で「改善提案」と「実改修」を同じ自動化経路に入れると、責任境界が曖昧になる。

したがって、生成と改善の責務境界を明示する必要がある。

## Decision

レッスン生成および改善ループは Owner ローカル実行のみに限定する。

- Owner ローカル環境で生成、編集、レビュー、最終判断を行う。
- サーバーの責務は draft の受け入れ、version 管理、公開、履歴保持に限定する。
- サーバー側の自動化上限は「改善提案レポート生成」までとし、レッスン本文や構造の実改修は自動適用しない。
- サーバーはレッスン差分を保存できても、無人で publish してはならない。
- 生成結果の取り込みは draft API もしくは管理 UI を通す。
- 他 ADR の非同期基盤やモデルルーティングは、この境界を越えない。つまり、ADR-0001 と ADR-0002 のイベントや jobs は、受け入れ、検証、レポート、公開補助に使うが、本文改修には使わない。

## Consequences

- 品質責任者が明確になり、MVP で最も重要な教材品質を人間中心に担保できる。
- サーバー側に広いツール権限や高額な生成処理を置かずに済む。
- 変更履歴、承認、公開の責務が整理され、事故時の原因切り分けが容易になる。
- 一方で、コンテンツ供給速度は Owner の稼働に依存する。
- 将来的に大規模運用へ進むには、レビュー基準、承認フロー、差分検証、自動テストを先に整える必要がある。

## Alternatives Considered

### サーバー側でレッスン生成まで自動化する

不採用。生成品質のばらつき、コスト暴走、外部ツール権限の集中が一度に発生する。MVP で欲しいのは速度よりも教材品質と責任境界の明確さである。

### サーバー側で改善提案から実改修まで自動適用する

不採用。提案と改修は責任の重さが異なる。提案生成までは補助だが、実改修は教材内容そのものを変えるため、人手レビューなしに進めるべきではない。

### ローカル生成とサーバー自動改修を混在させるハイブリッド方式

不採用。どこからが人間の確定判断で、どこからが自動変更かが曖昧になる。障害調査、品質保証、公開承認の経路も複雑化し、MVP に対して過剰設計である。

## Phase 8 Boundary Notes

- `lesson-factory/` の YAML / Markdown を Git の唯一 SoT とし、Supabase の `lesson_atoms` / `lesson_atom_versions` は sync 結果の配信用キャッシュとして扱う。
- 旧 canonical 系 (`lesson_identities`, `plans_v2`, `plan_nodes`) と旧 TypeScript curriculum 実装は Phase 8 で完全削除した。
- サーバー側の LLM 呼び出しは lesson 生成・改善に使わない。改善提案レポートもルールベース生成でコスト 0 とする。
- Atom 改善は Owner ローカルのみで実行し、Claude Code / Gemini サブスクを使う。サーバーは自動改修も自動 publish も行わない。
