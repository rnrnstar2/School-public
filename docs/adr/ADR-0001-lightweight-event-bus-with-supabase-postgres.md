# ADR-0001: Supabase Postgres をイベントバックボーンとして採用する

- Status: Accepted
- Date: 2026-04-08

## Context

MVP では、実行系、生成系、改善系を疎結合に保ちつつ、イベント駆動でつなぐ必要がある。一方で合意済みの制約は明確である。

- AWS は一切使わない。
- MVP 段階で採用する基盤は Supabase と Sentry に限定する。
- 実運用で増えるコンポーネントは最小限にし、Owner ローカル実行を含む開発体験を壊さない。
- イベントは後追い調査、再送、監査ができる形で保持したい。

このため、単なるリアルタイム通知だけでは不十分であり、永続化されたイベント正本と低遅延通知の両方が必要になる。

## Decision

MVP のイベントバックボーンは Supabase Postgres 上に実装する。

- ドメイン更新と同一トランザクションでイベントテーブルに 1 行書き込む。
- イベントテーブルをイベント正本とし、再送、追跡、リプレイの起点にする。
- 低遅延起動には `LISTEN/NOTIFY` を使うが、通知は最適化経路として扱い、信頼の起点にはしない。
- 取りこぼし回収、定期ポーリング、遅延実行には `pg_cron` を使う。
- ペイロードは `event_type`、`aggregate_id`、`schema_version`、`occurred_at`、`payload_json` を基本とする。
- Consumer はイベント ID もしくは `job_id` を使って冪等に処理する。

要するに、正本はイベントテーブル、即時性は `LISTEN/NOTIFY`、回復性は `pg_cron` で担保する。

## Consequences

- Supabase だけでイベント連携が完結し、MVP の運用面がかなり軽くなる。
- SQL でイベント履歴を直接追えるため、障害調査と監査が容易になる。
- アプリケーションコードは「書き込みとイベント発行を同時に確定する」前提で組める。
- `LISTEN/NOTIFY` だけに依存しないため、接続断や通知取りこぼしに対して回復できる。
- 一方で、大量ファンアウト、高スループット、複雑なストリーム処理には向かない。
- 将来、イベント量や Consumer 数が増えた場合は、イベントテーブルを outbox として維持したまま外部バスへ段階的に切り出せる。

## Alternatives Considered

### EventBridge

不採用。AWS 不使用方針に反するうえ、MVP のためにクラウド基盤を新設することになる。イベントスキーマ管理、IAM、配信失敗時の運用を追加で持つコストに対し、現時点のイベント量は小さい。

### Google Pub/Sub

不採用。AWS 回避は満たしても、Supabase 以外に常時運用するメッセージ基盤が増える。認証、課金、再試行、ローカル開発時の再現が別系統になり、MVP の単純さを崩す。

### Kafka

不採用。ブローカー運用、トピック設計、パーティション、Consumer Group、保持期間管理まで含めて明らかに過剰である。MVP で必要なのはイベント監査と軽い非同期連携であり、ストリーム基盤ではない。

### Inngest

不採用。イベント受け渡しとワークフロー意味論を外部サービスに寄せることになり、Postgres に既にある状態遷移と責務が二重化する。外部往復も増え、障害点が 1 つ増える。
